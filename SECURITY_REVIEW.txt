CRIT-1: Hardcoded Private Key with Real Funds Committed to Repository
----------------------------------------------------------------------
File: SecondSet/SecondSet/secondset/.env — Line 48
Class: Hardcoded Secret / Credential Exposure
Detail:
  TEST_SIGNER_PRIVATE_KEY="0x2dc4aae584c5ea8d6906b2ce5e9d2489cbd394dad547d0b0153d39492bbfa3c2"
  The .env file is present on disk in the repo. If it was ever accidentally

CRIT-5: Join Token Never Validated During Keygen Session Join
-------------------------------------------------------------

Detail:

CRIT-7: ECDSA Nonce Scalar Shared to Coordinator in Plaintext (Protocol Flaw)
------------------------------------------------------------------------------
Files:
  secondset-mobile-signer/mobile-signer/src/services/TSS/SigningClient.ts — Lines 184-194
  secondset-mobile-signer/coordinator/src/services/WebSocketManager.ts — Lines 550-570
Class: Cryptographic Protocol Flaw — Key Material Leaked to Coordinator
Detail:
  During ECDSA signing, each mobile signer broadcasts its ephemeral nonce
  scalar k_i unencrypted over the WebSocket to the coordinator:
    payload: JSON.stringify({
      noncePoint: this.myNoncePoint,
      nonce: this.myNonce,  // <-- raw scalar k_i, sent in plaintext
    })
  The coordinator is supposed to be a "stateless message router" with no key
  material, but it receives both signers' nonce scalars k_1 and k_2.
  From partial signatures s_i and combined k = k_1 + k_2, an attacker can
  compute the full private key: privkey = (s * k - e) / r mod n.
  This completely breaks the claim that "private keys never leave the device."
Impact: A compromised coordinator can extract the full private key from a
  single signing session. This eliminates the entire threshold security model.
  This is the most architecturally severe finding in this review.
Fix:
  - Use a proper 2-party ECDSA protocol (GG18, GG20, or DKLS19) where nonce
    scalars never leave each party.
  - Each party generates k_i * G = R_i, the combined R = R_1 + R_2 is computed
    via EC point addition (no scalar sharing needed).
  - The coordinator should only ever see public points (R_i), not scalars (k_i).
Remediated: [ ]

HIGH-3: WebSocket Session Isolation Bypass — Cross-Session Message Injection
----------------------------------------------------------------------------
File: secondset-mobile-signer/coordinator/src/services/WebSocketManager.ts — Lines 457-478, 550-570
Class: Broken Access Control / Session Isolation
Detail:
  In handleKeygenRound and handleSignRound, when routing to a specific
  participant (message.to_participant !== '*'), the coordinator routes to
  this.clients.get(message.to_participant) WITHOUT verifying that the target
  belongs to the same session. Also, message.from_participant is taken from
  the message payload (not from ws.participantId), allowing identity spoofing.
Impact: Cross-session message injection — a malicious participant can corrupt
  DKG round messages in another organization's vault creation ceremony.

  - Reject messages where message.from_participant !== ws.participantId (use JWT).
  - Use ws.participantId (from JWT) as the authoritative identity, not client field.

HIGH-4: Race Condition in Sign Complete Handler — Double Webhook Possible
------------------------------------------------------------------------
File: secondset-mobile-signer/coordinator/src/services/WebSocketManager.ts — Lines 573-790
Class: Race Condition / Business Logic
Detail:
  handleSignComplete: updates partial sig → queries count → if count >= required,
  sends webhook. There is no DB-level lock between the query and webhook send.
  Two simultaneous sign_complete messages can both see count >= required and
  both fire the webhook.
Impact: Potential double-broadcast of a signed transaction. The web app's
  idempotency check (updateMany where txHash: null) partially mitigates this.
Fix:
  - Use DB-level lock: UPDATE signing_sessions SET status = 'aggregating'
    WHERE status = 'in_progress' RETURNING session_id
  - Only the handler that successfully transitions to 'aggregating' fires webhook.
Remediated: [ ]

MED-8: EVM txDigest Binding Mismatch Between Approve and Release
----------------------------------------------------------------
Files:
  SecondSet/SecondSet/secondset/src/app/api/requests/[id]/approve/route.ts — Lines 111-113
  SecondSet/SecondSet/secondset/src/app/api/requests/[id]/release/route.ts — Lines 210-221
Class: Inconsistent Cryptographic Digest / Weak Binding
Detail:
  At approve time: txDigest = SHA256(serializeTransactionForDigest(evmTx))
  At release time: signingDigest = keccak256(serializeTransaction({ ... }))
  The release JWT verifies SHA256 digest matches, but mobile devices sign the
  keccak256 digest computed fresh from stored JSON fields. If stored JSON fields
  are malleable, the signed transaction could differ from what the approver saw.
Fix:
  Store the keccak256 signing digest at approve time and verify it at release time,
  binding approval directly to the bytes that will be signed.
  Or store and compare the full RLP-serialized transaction.
